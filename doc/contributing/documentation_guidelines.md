# 文档规范标准

## 1. 目录结构

```
doc/
├── guide/           # 指南文档
│   ├── quickstart.md        # 快速开始
│   ├── installation.md      # 安装指南
│   ├── configuration.md     # 配置指南
│   └── workflow.md          # 工作流指南
├── api/             # API 文档
│   ├── agent.md             # Agent 模块 API
│   ├── llms.md              # LLM 模块 API
│   ├── tools.md             # 工具模块 API
│   └── utils.md             # 工具函数 API
├── examples/        # 示例文档
│   ├── basic_usage.md       # 基本用法示例
│   ├── advanced_usage.md    # 高级用法示例
│   └── custom_workflow.md   # 自定义工作流示例
├── contributing/    # 贡献指南
│   ├── code_of_conduct.md   # 行为准则
│   ├── pull_request.md      # PR 指南
│   └── documentation_guidelines.md # 本文档
├── faq/             # 常见问题

│   ├── technical.md         # 技术问题

├── start.png        # 启动图

```

## 2. 命名规范

### 文件名
- 使用小写字母和连字符 `-` 分隔单词
- 文件名应简洁明了，反映文档内容
- 避免使用特殊字符和空格

### 目录名
- 使用小写字母和单数形式
- 目录名应反映其包含的文档类型

## 3. 内容格式

### 标题层级
- 使用 Markdown 标题语法
- 标题层级应清晰，从 `#` 开始，最多使用到 `####`

### 代码块
- 使用 Markdown 代码块语法
- 为代码块指定语言，如 ```python
- 代码应包含必要的注释和说明

### 链接
- 内部链接使用相对路径
- 外部链接使用完整 URL
- 图片链接应使用相对路径

### 表格
- 使用 Markdown 表格语法
- 表格应包含标题行
- 表格内容应对齐

## 4. 文档类型

### 指南文档
- 面向用户的使用指南
- 应包含详细的步骤说明和示例
- 应覆盖常见的使用场景

### API 文档
- 面向开发者的接口文档
- 应包含函数签名、参数说明、返回值和示例
- 应覆盖所有公共 API

### 示例文档
- 展示如何使用项目的示例
- 应包含完整的代码示例和说明
- 应覆盖不同的使用场景

### 贡献指南
- 面向贡献者的指南
- 应包含代码规范、PR 流程和文档规范
- 应帮助新贡献者快速上手

### 常见问题
- 解答用户常见的问题
- 应分类组织问题
- 应包含问题描述和解决方案

## 5. 最佳实践

### 文档更新
- 代码变更时应同步更新相关文档
- 版本发布时应更新版本相关文档
- 定期检查文档的准确性和完整性

### 文档审查
- 提交 PR 时应包含文档更新
- 文档变更应经过审查
- 确保文档语言清晰、准确

### 国际化
- 重要文档应提供中英文版本
- 英文文档应放在根目录
- 中文文档应放在 doc 目录

## 6. 工具推荐

- 使用 Markdown 编辑器编写文档
- 使用语法检查工具确保文档格式正确
- 使用版本控制系统管理文档变更

## 8. 版本控制

- 文档应与代码版本保持一致
- 重要的文档变更应在版本发布说明中提及
- 旧版本的文档应存档
